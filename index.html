Managementsamenvatting

Tien hefbomen, geordend naar impact

Elk woord dat je model in- en uitgaat is een token, en tokens zijn geld. Het grootste deel van dat verbruik is verspilling die je kunt wegsnijden. Deze tien leveren het meeste op — de eerste twee zijn vrijwel altijd de grootste winst.

De volledige gids

25 technieken in vijf categorieën

Van snelle prompt-ingrepen tot architectuurkeuzes. Begin bovenaan — prompt engineering en context management vragen de minste moeite voor de meeste winst. Bij elke techniek staat een "Zo pas je het toe"-blok: een concreet codevoorbeeld of stappenplan waarmee je vandaag nog kunt beginnen.

Categorie 01

Prompt engineering

Snelste ingrepen. Geen infrastructuur nodig — puur hoe je de prompt opbouwt.

1Trim de systeemprompt

Verwijder herhaalde regels, verbale beleefdheid en overbodige constraints. Vervang alinea's door bondige bullet-directieven; gebruik delimiters in plaats van proza.

Zo pas je het toe
# Voor — 40+ tokens, veel ruis
"Je bent een vriendelijke, behulpzame AI-assistent. Help de
gebruiker zo goed mogelijk. Wees altijd beleefd en netjes..."

# Na — zelfde gedrag, ~10 tokens
"Rol: support-agent. Taal: NL. Toon: bondig, correct."
Vaak −20 tot −40% promptgrootte

2Snoei few-shot voorbeelden

Elk voorbeeld betaal je bij iedere call opnieuw. Zodra het model betrouwbaar werkt, kun je er drastisch in snoeien.

Zo pas je het toe
  1. Meet de kwaliteit met 5 voorbeelden als nulmeting.
  2. Halveer naar 2, draai je testset opnieuw. Gelijk gebleven? Ga door.
  3. Test met 1 of 0. Zakt de kwaliteit? Eén stap terug.
  4. Zit gedrag écht vast? Verplaats de voorbeelden naar fine-tuning — dan betaal je ze nooit meer per call.
Honderden–duizenden tokens per call

3Stuur op korte output

Output is de dure kant (±4–5× input-prijs). Eén zin in je systeemprompt begrenst 'm.

Zo pas je het toe
system = "Antwoord in max 3 zinnen. Geen inleiding,
geen herhaling van de vraag, geen afsluitende samenvatting."
Snijdt direct in output-tokens

4Structuur boven proza

Scheid context, instructie en data met tags. Betrouwbaarder én je kunt de omlijstende taal wegsnijden.

Zo pas je het toe
<context>{documenten}</context>
<vraag>{vraag}</vraag>

Beantwoord de vraag alleen op basis van context.
Aanbevolen door Anthropic
Categorie 02

Context management

Bepaal wat er überhaupt de prompt in gaat. Vaak de grootste input-besparing.

5RAG in plaats van stuffing

Plak niet het hele document in de prompt. Haal alleen de top-k relevante stukken op via vector-similariteit.

Zo pas je het toe
# Haal 4 relevante stukken op i.p.v. 50 pagina's
stukken = vectorstore.similarity_search(vraag, k=4)
context = "\n\n".join(s.page_content for s in stukken)
Vaak 5–10× minder input

6Gesprek samenvatten

In lange chats groeit de context bij elke beurt. Comprimeer oude beurten tot een lopende samenvatting; houd alleen de recente letterlijk.

Zo pas je het toe
from langchain.memory import \
    ConversationSummaryBufferMemory

# Oude beurten worden auto-samengevat boven 1000 tokens
memory = ConversationSummaryBufferMemory(
    llm=llm, max_token_limit=1000)
Stopt lineaire groei

7Prompt-compressie — LLMLingua

Een klein model schrapt laag-informatieve tokens vóór het dure model ze ziet. Taak-agnostisch, plug-and-play op lange context.

Zo pas je het toe
from llmlingua import PromptCompressor
c = PromptCompressor()
# rate=0.5 → ongeveer 2× korter
korter = c.compress_prompt(context, rate=0.5)
2–20× minder tokens · Microsoft Research

8Gist tokens

Geavanceerder: leer het model een vaste prompt samen te persen tot enkele "gist"-tokens die je daarna cachet en hergebruikt.

Zo pas je het toe
  1. Kies de vaste instructie/preamble die bij elke call terugkomt.
  2. Fine-tune (of gebruik een gist-adapter) zodat die preamble in een paar tokens past.
  3. Cache de gist-tokens en zet ze vóór elke nieuwe input.
  4. Overweeg dit pas bij hoog volume — de winst (tot ~26×) betaalt de eenmalige trainingskost terug.
Tot ~26× · Stanford (arXiv:2304.08467)

9Dedupliceer context

Retrieval levert vaak overlappende of dubbele stukken. Strip ze eruit — goedkoop en verliesvrij.

Zo pas je het toe
# Gooi exacte duplicaten weg, behoud volgorde
gezien = set()
uniek = [s for s in stukken
         if not (s in gezien or gezien.add(s))]
Verliesvrije winst
Categorie 03

Caching & infrastructuur

Betaal niet twee keer voor hetzelfde. Grootste ROI van de hele lijst.

10Anthropic prompt caching

Markeer je vaste prefix (systeemprompt, tools, grote docs) met cache_control. Cache-reads kosten nog maar 10% van normale input.

Zo pas je het toe
system=[{
    "type": "text",
    "text": GROTE_SYSTEEMPROMPT,
    "cache_control": {"type": "ephemeral"}
}]  # vanaf de 2e call binnen 5 min: ~90% goedkoper
−90% op gecachete tokens · ✅ 13-07-2026

11OpenAI prompt caching

Volledig automatisch — je hoeft niets te markeren. De enige regel: houd je vaste content vooraan en variabele input achteraan.

Zo pas je het toe
  1. Zet systeemprompt + vaste context altijd bovenaan (identieke prefix).
  2. Zet de wisselende gebruikersinput onderaan.
  3. Zorg dat de vaste prefix > 1.024 tokens is — dan cachet OpenAI 'm vanzelf.
  4. Klaar. Cached input is ~90% goedkoper, retentie tot 24u (2026-update).
Stapelt met Batch tot ~75% · ✅ 13-07-2026

12Batch-API's

Werk dat niet direct terug hoeft (evals, bulk-classificatie, nachtelijke generatie) draai je async voor de helft van de prijs.

Zo pas je het toe
# Anthropic — 50% korting, klaar binnen 24u
client.messages.batches.create(requests=[
    {"custom_id": "rij-1", "params": {...}},
    # ... duizenden rijen in één job
])
−50% · beide aanbieders · ✅

13Semantic caching (GPTCache)

Cache op betekenis, niet op exacte tekst. "Wat kost verzending?" en "Hoeveel is de verzendkosten?" krijgen hetzelfde cache-antwoord — de LLM wordt overgeslagen.

Zo pas je het toe
from gptcache.adapter import openai
# Drop-in: zelfde call, maar bijna-identieke
# vragen komen uit de cache i.p.v. het model
openai.ChatCompletion.create(model=..., messages=...)
Slaat de LLM volledig over

14KV-cache optimalisatie

Voor wie zelf host: vLLM hergebruikt gedeelde prompt-prefixes over requests heen. Bespaart geheugen en verhoogt throughput — één vlag.

Zo pas je het toe
# Prefix-caching aan bij het opstarten van vLLM
vllm serve mijn-model --enable-prefix-caching
Throughput ↑ · self-host · docs.vllm.ai
Categorie 04

Output-optimalisatie

De dure kant. Elke output-token telt ±4–5× zwaarder dan input.

15max_tokens + stop-sequences

Zet een harde bovengrens op de generatie en laat 'm vroeg stoppen. Je betaalt nooit meer voor gebazel dan nodig.

Zo pas je het toe
client.messages.create(
    ...,
    max_tokens=150,              # harde cap
    stop_sequences=["\n\n"])   # stop bij lege regel
Geen betaling voor overtollige tokens

16Schema-gebonden output

Dwing JSON volgens een schema af. Geen "Zeker! Hier is…"-narratie meer, compacte output, en geen re-prompts bij kapotte JSON.

Zo pas je het toe
from pydantic import BaseModel
class Ticket(BaseModel):
    prioriteit: str
    categorie: str

resp = client.responses.parse(
    ..., text_format=Ticket)   # gegarandeerd valide
Minder output + betrouwbaar parsen

17Enums boven proza

Bij classificatie heb je geen zin nodig, alleen een label. Eén token in plaats van dertig.

Zo pas je het toe
system = "Classificeer de review. Antwoord met exact"
"één woord: POSITIEF | NEGATIEF | NEUTRAAL."
Tot 30× minder output-tokens

18Streaming + vroeg stoppen

Stream het antwoord en stop met consumeren zodra je hebt wat je nodig hebt — bijvoorbeeld het eerste complete JSON-object.

Zo pas je het toe
for chunk in stream:
    buffer += chunk
    if json_compleet(buffer):
        break   # rest niet meer ophalen
Betaal alleen voor wat je gebruikt
Categorie 05

Architectuur & ontwerp

Structurele keuzes. Meer werk vooraf, blijvende besparing achteraf.

19Model-routing / cascade

Niet elke taak heeft je duurste model nodig. Bepaal de moeilijkheid en route: simpel → klein model, moeilijk → vlaggenschip.

Zo pas je het toe
# Simpele heuristiek als startpunt
model = ("claude-haiku" if len(vraag) < 200
         and not heeft_code(vraag)
         else "claude-opus")
# Later: laat RouteLLM de router leren
−50 tot −85% op gemengd werk

20Distillatie / fine-tuning

Fine-tune een klein model zodat je de lange instructie-preamble volledig kunt schrappen — het gedrag zit dan in de gewichten.

Zo pas je het toe
  1. Verzamel 100–1.000 goede input/output-paren (bijv. uit je huidige logs).
  2. Fine-tune een klein model op die paren.
  3. Verwijder de lange preamble + few-shot — die zitten nu in het model.
  4. Eenmalige trainingskost, daarna permanent minder tokens per call.
Blijvende besparing per call

21Tool-hygiëne

Elk tool-schema wordt bij iedere call meegestuurd. Geef het model alleen de tools die bij de huidige stap passen.

Zo pas je het toe
# Niet alle 30 tools elke call — filter per stap
tools = [t for t in ALLE_TOOLS
         if relevant(t, huidige_stap)]
# En: houd elke tool-beschrijving kort
Minder vaste overhead per call

22Multi-agent met smalle context

Geef elke subagent alleen de plak context die hij nodig heeft — niet de volledige geschiedenis. Laat enkel de conclusie terugvloeien.

Zo pas je het toe
  1. Splits de taak in deeltaken met een duidelijke, smalle scope.
  2. Geef elke subagent alleen zíjn stukje input, niet het hele dossier.
  3. Laat de subagent enkel de conclusie terugsturen, niet zijn werkcontext.
  4. De orchestrator houdt zo een kleine, goedkope context.
Kleine context per call

23Map-reduce samenvatten

Voor een enorm document: vat de stukken parallel samen (map) en combineer daarna de samenvattingen (reduce). Nooit één gigantische context.

Zo pas je het toe
# MAP — elk stuk apart (kleine calls, parallel)
delen = [llm(f"Vat samen: {c}") for c in chunks]
# REDUCE — combineer de samenvattingen
eind = llm(f"Combineer tot één geheel: {delen}")
Begrenst context per call

24Templating / hergebruik

Deterministische output (standaard e-mails, vaste rapporten) hoeft niet door de LLM. Template met code, roep het model alleen aan voor het variabele stukje.

Zo pas je het toe
# Geen LLM-call voor de vaste romp
mail = TEMPLATE.format(naam=naam, bedrag=bedrag)
# LLM alleen voor het echt variabele deel
mail += llm(f"Schrijf 1 persoonlijke zin voor {naam}")
0 tokens voor het vaste deel

25Comprimeer opgehaalde chunks

Combineer RAG met compressie: draai na retrieval een goedkope compressiestap over de chunks vóór ze het dure model in gaan.

Zo pas je het toe
# RAG → compressie → pas dán het dure model
stukken = vectorstore.similarity_search(vraag, k=6)
stukken = [c.compress_prompt(s, rate=0.5)
           for s in stukken]
Stapelt op RAG-winst
